# 中文文档治理宪章

日期：`2026-04-19`
状态：`生效中`

## 1. 宪章目的

`astock_lifespan-alpha` 从现在开始采用文档先行、执行留痕、结论可追溯的治理方式。

这不是为了补形式，而是为了防止两种已经出现的问题继续扩大：

- 代码先改完，但没人说得清楚这次改动到底冻结了什么边界。
- 执行已经发生，但没有 `record / evidence / conclusion`，导致后续无法判断哪些结论已成立。

## 2. 正式文档分层

正式文档只认四层：

### `docs/01-design/`

回答“为什么这样设计”。

这里放：

- 系统边界
- 模块职责
- 分阶段设计理由
- 当前生效的治理原则

### `docs/02-spec/`

回答“正式契约是什么”。

这里放：

- 总体重构方案
- 输入输出契约
- 模块规格
- 文档治理与执行准入规则

### `docs/03-execution/`

回答“这次到底做了什么、怎么验证、结论是什么”。

这里必须形成闭环：

`card -> evidence -> record -> conclusion`

### `docs/04-reference/`

当前仓库暂未建立此层。

如果后续需要引入旧系统资料、外部材料、图稿来源，统一落到 `04-reference`，且只能作为参考输入，不能直接替代正式设计与规格。

## 3. 强制规则

从本宪章生效后，任何正式重构任务都必须遵守：

1. 先有 `design/spec`，再进入 `execution`。
2. 进入执行后必须生成 card。
3. 有命令、有测试、有验证，就必须写 evidence。
4. 有实际施工过程，就必须写 record。
5. 一个执行批次结束，必须写 conclusion，明确“接受 / 有条件接受 / 拒绝”。
6. 所有正式文档一律中文。

## 4. 唯一例外

本仓库的“阶段一基础重构”发生在治理恢复之前，因此允许进行一次性补记。

但补记只适用于这一批次，不构成以后继续先施工后补文档的理由。

## 5. 对后续阶段的要求

从第二阶段开始，任何任务如果缺少以下输入，不应直接编码：

- 对应 design
- 对应 spec
- 对应 execution card

如果只是探索、对账或问题定位，可以先写短说明；一旦进入正式实现，就必须回到标准闭环。

## 6. 治理目标

本宪章希望形成的不是“文档很多”，而是三件事：

1. 每个阶段都有清晰的边界和冻结点。
2. 每次执行都能回溯到证据和结论。
3. 后续任何人接手时，都能从文档直接判断系统当前真实状态，而不是猜。

## 7. 2026-04-24 补记：状态入口维护规则

当某个系统地基、模块 gate 或正式语义被登记为 `放行 / 冻结 / 拒绝 / 待修` 时，不能只更新 execution 结论文件。

必须同步维护状态入口：

- `docs/README.md`：让读者能从文档首页看见当前真实状态
- `docs/03-execution/README.md`：让执行区能定位最新 card 与结论
- `docs/03-execution/00-conclusion-catalog-20260419.md`：让结论目录保持可追溯
- 必要时补入 `docs/02-spec/`：当执行结论固化为长期语义契约时，必须形成正式规格

`MALF` Card 62 是本规则的当前样例：执行结论确认 `malf = 放行` 后，状态入口、Canon 规格与结论目录必须同步更新，避免后来者只读旧阶段文档而误判系统地基仍未成立。
