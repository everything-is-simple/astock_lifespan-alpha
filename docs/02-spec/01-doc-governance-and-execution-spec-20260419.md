# 文档治理与执行闭环规格

日期：`2026-04-19`
状态：`冻结`

## 1. 目的

本文档把 `astock_lifespan-alpha` 的文档治理从“口头要求”变成正式规格。

目标不是增加写作负担，而是明确：

- 什么文档才算正式输入。
- 什么条件下任务可以进入执行。
- 执行结束后必须留下什么痕迹。

## 2. 正式文档流

本仓库的正式文档流固定为：

```text
design -> spec -> execution
```

其中执行层必须继续展开为：

```text
card -> evidence -> record -> conclusion
```

含义如下：

- `design`：解释设计理由、边界与原则。
- `spec`：冻结正式输入输出、约束与验收口径。
- `card`：定义本次执行批次的目标、范围、任务切片与收口标准。
- `evidence`：保留命令、测试、关键输出与产物路径。
- `record`：描述实际施工过程、偏差与备注。
- `conclusion`：给出正式裁决与影响范围。

## 3. 目录约束

### 3.1 `docs/01-design/`

允许放：

- 设计说明
- 治理宪章
- 模块边界设计
- 阶段设计说明

### 3.2 `docs/02-spec/`

允许放：

- 总方案
- 模块规格
- 文档治理规格
- 图版规格说明

### 3.3 `docs/03-execution/`

根目录只放：

- `card`
- `conclusion`
- `README`
- 模板文件
- 执行纪律
- 结论目录

### 3.4 子目录约束

- `evidence` 只能放到 `docs/03-execution/evidence/`
- `record` 只能放到 `docs/03-execution/records/`

任何把 `*-evidence-*` 或 `*-record-*` 直接放进 `docs/03-execution/` 根目录的做法，都视为治理违规。

## 4. 执行准入规则

正式实现任务进入执行前，至少应具备：

1. 对应 design 文档
2. 对应 spec 文档
3. 已创建 execution card

若暂时只是探索、核对或问题定位，可以不立 card；但一旦进入正式代码落地，必须补齐上述输入。

## 5. 执行批次最小交付

任何一个正式执行批次，至少需要产出：

1. 一张 card
2. 一份 evidence
3. 一份 record
4. 一份 conclusion

缺任何一项，都不能视为执行闭环完成。

## 6. 状态定义

建议使用以下状态：

- `草稿`：文档已创建，尚未执行或尚未冻结
- `执行中`：已开始施工，但尚未完成闭环
- `已接受`：本批次交付被正式接受
- `有条件接受`：本批次可继续向下游释放，但仍有未清债项
- `已拒绝`：实现或结论不被接受，需要重开 card

## 7. 编号规则

执行文档编号与批次编号对齐，例如批次 `01`：

- `01-...-card-20260419.md`
- `01-...-evidence-20260419.md`
- `01-...-record-20260419.md`
- `01-...-conclusion-20260419.md`

同一批次的四类文档必须共享同一个编号。

## 8. 中文要求

本仓库正式文档统一使用中文。

允许保留必要英文术语、表名、字段名、模块名和 runner 名，但正文叙述、结论、范围说明、执行说明必须是中文。

## 9. 补记规则

当前仓库仅允许对“阶段一基础重构”进行一次性补记，原因是该批次代码已在治理恢复前落地。

从阶段二开始：

- 不再允许默认先落代码后补治理文档。
- 如确实出现紧急修复，必须在同日内补齐闭环文档。

## 10. 验收含义

当一个批次被标记为“已接受”时，代表：

- 范围内交付已经完成
- 关键验证已经留痕
- 当前结论可作为后续阶段输入

但这不意味着后续实现不能继续迭代；它只意味着本批次冻结的边界已经成立。
